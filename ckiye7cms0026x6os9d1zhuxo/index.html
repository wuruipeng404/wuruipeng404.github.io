关于RabbitMQ消息队列的可靠性传输

发表于 2020-10-28 分类于 消息队列 阅读次数：

在使用消息队列的过程中如何保证消息的可靠性呢？
这个问题其实包含了几方面的小问题。

  1. 数据丢失问题
  2. 数据幂等性的问题
  3. 重复消费的问题

我们知道在使用消息队列的时候是必然有三个角色的，生产者，消费者，以及MQ本身。
那么数据丢失无非就以下几种情况

  1. 生产者传输给MQ 这个过程丢失数据，导致MQ未获取到消息
  2. MQ收到消息，但是自身宕机，把消息给弄丢了。
  3. MQ将消息传输给消费者，消费者接收到消息，但是还没处理 挂掉了。导致消息丢失。

解决办法：

  1. 第一种情况有两种解决办法

    1. 是在生产者开启MQ的事务机制，整个流程和数据库事务差不多，但是这种方法完全不推荐。首先事务机制会大幅降低MQ的吞吐量和性能。其次，我们用MQ本来就是追求一个异步的消息传输过程，当开启事务时，所有的一系列流程就变成了同步阻塞。
    2. 在生产者开启confirm模式(推荐)，开启confirm模式之后 每一条消息mq都会返回给你一个ack信息，用来通知你消息是发送成功还是失败。而且这整个过程是异步的。下面用go的代码做一下示例：
      package main

import (
  "fmt"
  "github.com/streadway/amqp"
  "log"
)

func main() {
  var (
    err  error
    conn *amqp.Connection
    ch   *amqp.Channel
    q    amqp.Queue
  )
  if conn, err = amqp.Dial("amqp://guest:guest@localhost:5672/"); err != nil {
    log.Fatalf("连接MQ失败:%s", err.Error())
  }

  defer func() {
    _ = conn.Close()
  }()

  // 打开一个channel
  if ch, err = conn.Channel(); err != nil {
    log.Fatalf("获取channel失败:%s", err.Error())
  }

  defer func() {
    _ = ch.Close()
  }()

  // 声名队列
  if q, err = ch.QueueDeclare(
    "hallo",
    false,
    false,
    false,
    false,
    nil,
  ); err != nil {
    log.Fatalf("声明队列失败:%s", err.Error())
  }

  confirm := ch.NotifyPublish(make(chan amqp.Confirmation))
  if err = ch.Confirm(false); err != nil {
    log.Fatalf("开启confirm模式失败:%s", err.Error())
  }

  go func() {
    for i := 0; i <= 10; i++ {
      if err = ch.Publish(
        "",
        q.Name,
        false,
        false,
        amqp.Publishing{
          ContentType: "text/plain",
          Body:        []byte(fmt.Sprintf("%d", i)),
        },
      ); err != nil {
        log.Printf("推送消息失败:%s", err.Error())
        continue
      }
    }
  }()

  for i := 0; i <= 10; i++ {

    ack := <-confirm
    if ack.Ack {
      // 消息发送成功
    } else {
      // 消息发送失败
      // 写你想要的逻辑
    }
    log.Println(ack.DeliveryTag)
  }
}

  2. 第二种情况，MQ消息丢失

    1. 这个时候就需要开启MQ的数据持久化模式了。

    2. 持久化包括三个部分：

      // 1. exchange 的持久化 
// 声明exchange 时将 durable设置为True
ch.ExchangeDeclare(
  "logs", // name
  amqp.ExchangeFanout, // kind 
  true, // durable
  false, // autoDelete
  false, // internal  是否为内置交换机
  false, // noWait
  nil,   // args
)

// 2. queue 持久化
// 同样的在声明时 将 durable设置为True
ch.QueueDeclare(
  "hallo", // name 队列名称
  true,  // durable 是否持久化
  false, // autoDelete 
  false, // exclusive 排他性
  false, // noWait 
  nil, // args
)

// 3. message 持久化
// 发送消息时将 delivery_mode 设置为2
ch.Publish(
  "logs", // exchange
  "",     // routing key
  false,  // mandatory 无法路由的消息是否返回处理
  false,  // immediate 是否对路由到无消费者队列的消息进行返回处理
  amqp.Publishing{
    ContentType: "text/plain",
    Body:        []byte("hallo world"),
    DeliveryMode: 2,
  },
)

      注意：

      • exchange 和 queue 如果一个为持久化一个不为持久化，那么他们之间就不能进行绑定
      • 如果都为持久化，那么绑定也是持久化的
      • 设定持久化之后是不允许修改的，唯一途径是删除重建。
      • 即使你做了持久化，也还是不能完全的保证 数据不丢失，比如MQ在持久化之前挂掉了，那当时内存的数据就真的丢失了，只是这种情况的概率非常小。
      • 这一部分还可以引申出 rabbitmq 搭建高可用集群这一话题，这里就不详细讲了。简单描述一下，rmq的集群模式分为两种，一种是普通模式，比如有两个节点，那么queue只需要存在于其中一个节点即可，每个节点的数据是不同的。第二种是镜像模式，节点与节点之间会主动同步数据，这种优点和缺点都是显而易见的。反正根据自己的实际情况来选择。当然了有钱直接阿里云他不香吗。

  3. 第三种情况就需要启用我们的ack机制了，就是当我们消费处理完毕的时候，去给mq返回一个ack确认信息。

    if msgs, err = ch.Consume(
  q.Name, // name
  "",     // consumer
  false,   // 是否自动ack
  false,   // exclusive 排他
  false,   // noLocal
  false,   // noWait
  nil,     // args
); err != nil {
  log.Fatalf("获取消费队列失败:%s", err.Error())
}

for d := range msgs {
  // 处理消息逻辑
  if err = d.Ack(false); err != nil {
    log.Printf("确认消息失败:%s", err.Error())
  }
  log.Printf("收到消息:%s", d.Body)
}
